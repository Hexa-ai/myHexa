> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Installer Playwright et écrire les 4 parcours E2E critiques (lien signé valide, lien expiré + recover, login + isolation tenant, accès cross-tenant refusé). Documenter la procédure de cutover pour chaque workflow n8n, à exécuter au rythme choisi par l'utilisateur.

**Architecture :** Tests Playwright dans `app/tests/e2e/`, configuration pour cibler une instance Supabase de test (ou la prod en lecture seule avec comptes dédiés). Cutover runbook = checklist par workflow, à dérouler lorsque l'utilisateur valide l'équivalent Vue.

**Tech Stack :** Playwright + GitHub Actions (manuel/tag) + procédures manuelles n8n (`mcp__n8n-mcp__n8n_get_workflow`, `n8n_update_partial_workflow`).

**Prérequis :**
- P1 à P4 terminés et déployés sur Netlify
- Au moins 2 companies de test en base avec membres associés (créées en P2 Task 10)
- Un device test avec token et un report test avec link signé (à créer si pas déjà fait)

### Task 6 : Test E2E 4 — Cross-tenant denied

**Files:**
- Create: `app/tests/e2e/04-cross-tenant-denied.spec.ts`

L'idée : member A tente d'éditer un device de Company B via une requête SQL côté client. Doit échouer (RLS bloque).

- [ ] **Step 6.1 : Écrire le test**

```ts
import { test, expect } from '@playwright/test'
import { login } from './helpers/auth'
import { MEMBER_A, COMPANY_B_DEVICE_ID } from './helpers/fixtures'

test('member A cannot read company B devices via direct supabase call', async ({ page }) => {
  await login(page, MEMBER_A.email, MEMBER_A.password)

  // Évalue dans le contexte de la page : tente un select direct sur un device
  // qui appartient à Company B → RLS doit filtrer ce row (data vide, pas d'erreur)
  const result = await page.evaluate(async (deviceId) => {
    const { supabase } = await import('/src/lib/supabase.ts')
    const { data, error } = await supabase
      .from('devices')
      .select('id')
      .eq('id', deviceId)
    return { data, error: error?.message ?? null }
  }, COMPANY_B_DEVICE_ID)

  expect(result.error).toBeNull()
  expect(result.data).toEqual([]) // RLS filtre, aucun row renvoyé
})
```

**Note :** `page.evaluate` avec import dynamique fonctionne en dev (Vite sert les modules). En CI sur le build prod, il faudra adapter (passer par un endpoint debug ou par les `<RouterLink>` admin).

Variante plus simple si le `evaluate` cause problème : tenter un `UPDATE` côté SQL via les DevTools, et vérifier qu'aucune row n'est affectée (la table ne change pas).

### Task 7 : Workflow GitHub Actions pour les E2E

**Files:**
- Create: `.github/workflows/e2e.yml`

L'E2E ne tourne **pas** sur chaque PR (trop lent). On le déclenche manuellement ou sur tag, contre le déploiement Netlify de preview.

- [ ] **Step 7.1 : Créer le workflow**

```yaml
name: E2E

on:
  workflow_dispatch:
    inputs:
      base_url:
        description: 'URL cible (e.g. https://deploy-preview-xxx.netlify.app)'
        required: true

jobs:
  e2e:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: app
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
          cache-dependency-path: app/pnpm-lock.yaml

      - run: pnpm install --frozen-lockfile
      - run: pnpm exec playwright install --with-deps chromium

      - run: pnpm test:e2e
        env:
          E2E_BASE_URL: ${{ github.event.inputs.base_url }}
          E2E_MEMBER_A_PASSWORD: ${{ secrets.E2E_MEMBER_A_PASSWORD }}
          E2E_MEMBER_B_PASSWORD: ${{ secrets.E2E_MEMBER_B_PASSWORD }}

      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: app/playwright-report/
```
